## Tool Use 机制疑问

### 1. 工具调用的并发控制
- **疑问**: 当一次需要调用多个工具时，如何控制并发和依赖关系？
- **状态**: ✅ 已解决
- **完成日期**: 2026-03-13
- **优先级**: 中
- **解决方案**:
  - OpenClaw 支持在单个响应中调用多个工具（tool_calls 数组）
  - 工具调用按顺序执行，如果工具间有依赖关系，应分多次调用
  - 独立的工具可以一次性调用多个，提高效率
  - AI 根据工具返回结果决定是否需要继续调用其他工具

### 2. 工具调用的错误处理
- **疑问**: 工具调用失败时的最佳错误处理策略是什么？是否应该自动重试？
- **状态**: ✅ 已解决
- **完成日期**: 2026-03-13
- **优先级**: 高
- **解决方案**:
  - 工具调用失败时，错误信息会返回给 AI，由 AI 判断如何处理
  - 不建议盲目自动重试，应根据错误类型（临时性错误、权限错误、参数错误等）决定
  - 临时性错误（如网络超时）可以重试
  - 参数错误或权限错误应直接告知用户
  - 重大错误应中止流程并明确报告给用户

### 3. 工具输出的长度限制
- **疑问**: 工具返回的输出长度有限制吗？如何处理大输出？
- **状态**: ✅ 已解决
- **完成日期**: 2026-03-13
- **优先级**: 低
- **解决方案**:
  - 工具输出长度受上下文窗口限制
  - 长输出会被截断，输出中会标记 `[truncated: output exceeded context limit]`
  - 处理策略：使用 offset/limit 参数分批读取，或使用工具特定参数过滤输出
  - 示例：`read` 工具支持 offset 和 limit 参数，避免一次性读取大文件

### 4. 工具调用的上下文管理
- **疑问**: 多次工具调用时，如何保持上下文的一致性和连贯性？
- **状态**: ✅ 已解决
- **完成日期**: 2026-03-13
- **优先级**: 高
- **解决方案**:
  - OpenClaw 通过 conversation history 维护上下文
  - 每次工具调用的结果都会加入对话历史
  - AI 可以基于历史记录继续调用工具，实现多轮交互
  - 关键信息需要显式传递，不能依赖隐式状态
  - 建议在调用工具时保留足够的上下文信息（如文件路径、状态等）
