# 强制性路径验证脚本

> 确保所有文件操作都使用 `.learning/` 前缀

**版本**：1.0.0
**创建日期**：2026-03-13

---

## 🚨 使用方法

### 方法 1：在文件操作前验证

在执行任何 `write` 或 `edit` 操作前，先验证路径：

```bash
# 验证路径
FILE_PATH="<计划保存的文件路径>"
if [[ ! "$FILE_PATH" == .learning/* ]]; then
  echo "❌ 路径错误：必须使用 .learning/ 前缀"
  echo "实际路径：$FILE_PATH"
  # 退出，不继续操作
  return false
fi

# 验证通过，继续操作
return true
```

### 方法 2：使用完整路径

所有文件操作必须使用以 `.learning/` 开头的完整路径：

```bash
# 错误示例
write "bookmarks/test.md" "内容"  # ❌ 错误

# 正确示例
write ".learning/bookmarks/test.md" "内容"  # ✅ 正确
```

## 🎯 关键原则

### 原则 1：路径必须验证

任何文件操作前，必须先验证路径是否正确。

**验证标准**：
- [ ] 路径以 `.learning/` 开头
- [ ] 路径不包含 `../` 或其他相对路径
- [ ] 路径不是绝对路径（不使用 `~/`）

### 原则 2：验证失败必须停止

如果路径验证失败，必须：
1. **立即停止操作**
2. **报告明确的错误**
3. **不得尝试"修复"或使用替代路径**

### 原则 3：明确告知模型

在任务中明确告知路径规则，而不是依赖文档。

**示例**：
```markdown
## 任务要求

创建书签时，必须：

1. **验证路径**：确保使用 `.learning/` 前缀
2. **禁止其他路径**：不得使用 `bookmarks/`、`~/` 等其他前缀
3. **严格验证**：如果路径不正确，立即停止并报错

**违反后果**：
- 如果使用了非 `.learning/` 路径，技能执行失败
- 不会尝试"修复"或"纠正"
```

### 方案 B：使用 exec 脚本验证

创建一个脚本验证路径：

```bash
#!/bin/bash
# validate-path.sh

FILE="$1"

# 验证路径前缀
if [[ ! "$FILE" == .learning/* ]]; then
  echo "❌ 错误：路径必须使用 .learning/ 前缀"
  exit 1
fi

echo "✅ 验证通过"
exit 0
```

然后在操作前执行：

```bash
exec: bash scripts/validate-path.sh ".learning/bookmarks/test.md" && echo "验证通过，继续创建" || echo "验证失败，停止操作"
```
