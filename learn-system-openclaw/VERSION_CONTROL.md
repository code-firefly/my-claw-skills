## 🎯 版本号规则

### 版本号格式

```
MAJOR.MINOR.PATCH
```

- **MAJOR**：主版本号（0, 1, 2...）
- **MINOR**：次版本号（0, 1, 2...）
- **PATCH**：补丁版本号（0, 1, 2...）

### 版本类型定义

#### 0.x.x - 开发版本

**说明**：
- 尚未发布到 ClawHub
- 在开发中的版本
- 可能包含破坏性变更

**规则**：
- 主版本号固定为 `0`
- 次版本号递增：0.1.0, 0.2.0, 0.3.0...
- 每次功能更新或重构都递增次版本号

**示例**：
- 0.1.0 - 初始版本
- 0.2.0 - 第一次重构
- 0.3.0 - 功能更新

#### 1.x.x - 稳定版本

**说明**：
- 已发布到 ClawHub 的稳定版本
- 不包含破坏性变更
- 可能包含向后兼容的更新

**规则**：
- 主版本号固定为 `1`
- 次版本号递增：1.0.0, 1.1.0, 1.2.0...
- 每次功能更新或优化都递增次版本号

**示例**：
- 1.0.0 - 首个稳定发布版本
- 1.1.0 - 功能更新 A
- 1.2.0 - 功能更新 B

#### 2.x.x - 重大版本

**说明**：
- 包含破坏性变更或重大重构
- 不兼容之前的版本
- 需要迁移指南

**规则**：
- 主版本号递增：1.0.0 → 2.0.0
- 次版本号重置为 0
- 每次重大变更都递增主版本号

**示例**：
- 2.0.0 - 第一次重大重构
- 3.0.0 - 第二次重大重构

## 🔄 版本发布流程

### 开发阶段（0.x.x）

**流程**：
1. 在开发中频繁更新
2. 每次重构或功能更新都递增次版本号
3. 不发布到 ClawHub
4. 只在本地测试和验证

**何时发布**：
- 当功能稳定且经过充分测试
- 当达到发布标准（≥80/100 分）
- 当用户准备好发布

### 稳定版本（1.x.x）

**流程**：
1. 当达到发布标准，将版本号更新为 1.0.0
2. 发布到 ClawHub
3. 更新 GitHub Release
4. 更新文档和说明

### 重大版本（2.x.x）

**流程**：
1. 当有破坏性变更时，递增主版本号
2. 提供迁移指南
3. 发布到 ClawHub
4. 更新文档和说明

## 📝 版本号使用规则

### 何时递增 PATCH 版本

- 修复小 bug
- 修正文档错误
- 不影响功能的代码优化

### 何时递增 MINOR 版本

- 添加新功能
- 功能优化
- 代码重构（不破坏性）

### 何时递增 MAJOR 版本

- 破坏性变更
- 重大重构
- 不兼容的更新
