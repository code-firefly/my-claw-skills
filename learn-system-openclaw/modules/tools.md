## 📚 书签工具

### 创建书签

**执行步骤**：

#### 步骤 1：自动获取上下文
- 从 `PROGRESS.md` 获取当前"进行中"的课程
- 从课程的 `checklist.md` 获取当前学习位置（最后一个有勾选的章节）

#### 步骤 2：询问用户
- 疑问点描述（想要探索什么问题？）
- 探索目标（可选，系统可智能生成）

#### 步骤 3：智能生成内容
根据疑问点类型，自动生成探索目标和返回验证：

| 疑问类型 | 探索目标 | 返回验证 |
|----------|----------|----------|
| 命令相关 | 阅读文档、实践使用、记录技巧 | 熟练使用该命令 |
| 概念相关 | 阅读资料、整理笔记 | 能够解释概念 |
| 配置相关 | 查看文档、实践配置 | 能够独立配置 |
| 机制相关 | 阅读源码、实验验证 | 能够说明工作原理 |

#### 步骤 4：显示预览
```
╔═══════════════════════════════════════════════════════╗
║                    书签预览                           ║
╠═══════════════════════════════════════════════════════╣
║ 名称：深入理解 Tool Use 机制                          ║
║ 来源课程：ai-tools-fundamentals                       ║
║ 主线位置：完整模式 > 第一部分 > 架构与概念            ║
╠═══════════════════════════════════════════════════════╣
║ 疑问点：                                              ║
║   Tool Use 的具体实现原理是什么？                      ║
╠═══════════════════════════════════════════════════════╣
║ 探索目标：                                            ║
║   - [ ] 阅读官方 Tool Use 文档                        ║
║   - [ ] 查看 Tool Use 的源码实现                      ║
╠═══════════════════════════════════════════════════════╣
║ 返回验证：                                            ║
║   - [ ] 能够解释 Tool Use 的工作原理                  ║
╚═══════════════════════════════════════════════════════╝
```

#### 步骤 5：确认并保存
用户确认后，将书签添加到 `LEARNING_BOOKMARKS.md`

---

### 继续书签

**执行步骤**：

1. 读取 `LEARNING_BOOKMARKS.md`

2. 如果未指定书签名称：
   - 显示所有 🔵 探索中 和 🟡 待探索 的书签列表
   - 等待用户选择

3. 如果指定了书签名称：
   - 直接定位到该书签

4. 显示书签详情和探索笔记

5. 更新书签状态为 🔵 探索中

6. ⭐ **显示探索模式提示**：
   ```
   ════════════════════════════════════════════════════════
   🔵 书签探索模式已启动

   当前书签：{书签名称}
   疑问点：{疑问点描述}

   💡 提示：
   • 探索期间请专注于当前疑问
   • 完成后说"完成书签"返回主线
   ════════════════════════════════════════════════════════
   ```

7. 询问用户下一步：
   - 开始探索？
   - 查看探索笔记？
   - 修改探索目标？

---

### 完成书签

**执行步骤**：

1. 显示书签详情和返回验证清单

2. 询问确认：
   - 返回验证是否都已满足？
   - 确认标记为完成？

3. 用户确认后：
   - 标记书签为 🟢 已完成
   - 记录完成时间（YYYY-MM-DD）
   - 更新 `LEARNING_BOOKMARKS.md` 中的统计

4. ⭐ **显示主线返回信息**：
   ```
   ════════════════════════════════════════════════════════
   ✅ 书签已完成，正在返回主线学习...

   📍 主线位置：
      课程：{课程名}
      进度：{学习模式} > {章节}
      下一步：{下一学习项}

   继续主线学习？(y/n)
   ════════════════════════════════════════════════════════
   ```

---

### 删除书签

**执行步骤**：

1. 显示警告：此操作将删除书签及其所有探索笔记

2. 询问确认（需要输入书签名称确认）

3. 从 `LEARNING_BOOKMARKS.md` 中移除书签

4. 更新统计信息

## 🗂️ 知识缓存工具

### 初始化知识缓存

**触发方式**：
- "初始化知识缓存 `<课程名>`" / "创建知识缓存 `<课程名>`" / "新建缓存 `<课程名>`"
- "拉取资料 `<课程名>`" / "获取资料 `<课程名>`" / "下载文档 `<课程名>`"
- "初始化缓存" / "创建缓存"（询问课程名）

**执行步骤**：

1. 从配置获取课程资料来源

2. 拉取官方资料（使用 MCP 工具，优化参数：`retain_images=false`，`timeout=20`）

3. 创建课程知识缓存目录：
   ```
   XX-阶段名称/课程名/knowledge/
   ├── README.md           # 缓存说明
   ├── overview.md         # 概述内容
   ├── concepts.md         # 核心概念
   ├── guides.md           # 使用指南
   └── .metadata.json      # 缓存元数据
   ```

4. 结构化存储到缓存文件

5. 更新 `KNOWLEDGE_CACHE.md` 状态表

**优化参数说明**：
- `retain_images=false`：跳过图片下载，提升加载速度 50-70%
- `timeout=20`：20秒超时，适合大型文档

### 刷新知识缓存

**触发方式**：
- "刷新知识缓存" / "更新知识缓存" / "刷新缓存"
- "更新缓存" / "同步资料" / "刷新资料" / "更新文档"
- "刷新知识缓存 `<课程名>`" / "更新 `<课程名>` 缓存" / "`<课程名>` 刷新缓存"

**执行步骤**：

1. 显示当前缓存状态（从 `KNOWLEDGE_CACHE.md`）

2. 询问更新范围：
   - 1. 全部课程
   - 2. 指定课程

3. 拉取最新资料（使用优化参数：`retain_images=false`，`timeout=20`）

4. 更新缓存文件和 `.metadata.json`

5. 更新 `KNOWLEDGE_CACHE.md` 更新历史

**优化参数说明**：
- `retain_images=false`：跳过图片下载，提升加载速度 50-70%
- `timeout=20`：20秒超时，适合大型文档

## 📋 行为规范

### 书签探索行为规范

> **当书签状态为 🔵 探索中 时，AI 应遵循以下行为规范：**

#### 探索阶段

1. **每次查询/操作完成后**：
   - 显示探索结果摘要（简洁，不超过 5 个要点）
   - ⭐ **关键：主动询问** "还有其他疑问吗？"
   - 等待用户响应，不要自动继续或返回主线

2. **用户回应的处理**：
   - 用户提出新疑问 → 继续探索，记录到探索笔记
   - 用户说"没有了"/"没问题" → 提示："可以说'完成书签'返回主线"
   - 用户说"完成书签" → 调用完成书签流程
   - 用户沉默/不明确 → 保持探索中状态，等待用户主动

3. **输出格式示例**：
   ```
   📚 探索结果：Tool Use 机制

   核心要点：
   • Tool Use 通过 function calling 实现
   • AI 主动调用外部工具，而非被动响应
   • 支持多工具并行调用

   ❓ 还有其他疑问吗？
      - 有疑问 → 直接告诉我
      - 没有了 → 说"完成书签"返回主线
   ```

#### 禁止行为

- ❌ 不要在查询完成后自动说"让我们返回主线学习"
- ❌ 不要推测用户没有疑问
- ❌ 不要在没有用户确认的情况下结束探索

### 理论知识展示行为规范

> 在学习过程中展示官方文档、概念解释、使用指南等理论知识时，AI 应遵循分块展示和用户确认机制，确保用户有充分时间理解内容。

#### 适用范围

| 场景 | 示例 |
|------|------|
| 官方文档 | GitHub README、API 文档、技术规范 |
| 概念解释 | 术语定义、原理说明、架构描述 |
| 使用指南 | 安装步骤、配置说明、操作流程 |
| 理论知识 | 学习资料、教程内容、最佳实践 |

#### 分块展示策略

**触发条件**：
- 展示内容超过 15 行
- 内容包含多个独立的逻辑单元（如多个概念、多个步骤）

**分块原则**：
- 按逻辑单元分块（如按概念、按步骤、按小节）
- 每块不超过 15 行（不含确认提示）
- 优先在自然段落边界分块

**示例**：
```
第 1/3 块：核心概念
[展示前 15 行内容]

═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════

[用户确认后...]

第 2/3 块：实现原理
[展示接下来的内容]
```

#### 确认提示格式

```
═══════════════════════════════════
❓ 是否继续下一部分？
   - 回复任意内容继续
   - "跳过"进入下一步
   - "暂停"稍后继续
═══════════════════════════════════
```

#### 用户响应处理

| 用户输入 | AI 行为 |
|---------|---------|
| 任意非指令内容（如"继续"、"好的"、"y"） | 继续展示下一块 |
| "跳过" | 跳过剩余理论内容，直接进入下一步操作 |
| "暂停" | 停止展示，告知用户可回复"继续"恢复 |
| "完成" | 结束理论展示，进入下一步操作 |

#### 来源标注规范

| 内容类型 | 标注方式 | 示例 |
|----------|----------|------|
| 官方文档 | 文档名称 + 链接 | `来源：Claude Code 官方文档 https://code.claude.com/docs` |
| GitHub README | 仓库名 + 文件路径 | `来源：anthropics/claude-code/blob/main/README.md` |
| API 文档 | 文档标题 + URL | `来源：MCP SDK API Reference` |
| 缓存知识 | 缓存日期 | `来源：知识缓存 (2026-03-07)` |
| AI 生成内容 | 无需标注 | （直接展示） |

**标注格式**：
```
📚 来源：Claude Code 官方文档
https://code.claude.com/docs/getting-started

───────

[内容开始]
```

#### 禁止行为

- ❌ 不要在展示完理论内容后自动继续下一步
- ❌ 不要推测用户已经读完内容
- ❌ 不要在用户未确认的情况下进入下一步
- ❌ 不要一次性展示超过 15 行的理论内容
- ❌ 不要因为 `accept edits on` 模式开启而省略理论确认

## 注意事项

### 通用注意事项

1. **用户确认**：修改多个文件前，总是先显示预览并等待用户确认
2. **日期格式**：统一使用 YYYY-MM-DD 格式
3. **文件路径**：所有路径相对于仓库根目录
4. **错误处理**：如果文件不存在或课程未找到，给出明确的错误提示

### 书签相关

- 书签探索期间，AI 应专注于书签疑问，避免讨论主线内容
- 探索结果使用要点列表，保持简洁（避免占用过多上下文）
- 每次查询后必须主动询问"还有其他疑问吗？"

### 缓存相关

- 缓存策略：每次询问用户，而非自动判断过期，给予用户完全控制
- 内容获取优化：拉取网页内容时使用 `retain_images=false` 跳过媒体文件，提升加载速度
