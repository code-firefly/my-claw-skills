# 多代理团队编排 (Agent Team Orchestration)

运行具有清晰角色、结构化任务流和质量关口的团队生产手册。

## 快速开始：最小的2代理团队

一个构建者和一个审查者。最简单的有用团队。

### 1. 定义角色

```
编排者（你）—— 路由任务，跟踪状态，报告结果
构建代理       —— 执行工作，产出工件
```

### 2. 派生任务

```
1. 创建任务记录（文件、数据库或任务板）
2. 派生构建者，并提供：
   - 任务 ID 和描述
   - 工件输出路径
   - 移交说明（要生产什么，放在哪里）
3. 完成后：审查工件，标记完成，报告
```

### 3. 添加审查者

```
构建者生产工件 → 审查者检查 → 编排者交付或返回
```

这就是核心循环。下面的所有内容都在扩展这个模式。

## 核心概念

### 角色

每个代理都有一个主要角色。重叠会造成混乱。

| 角色 | 目的 | 模型指导 |
|------|---------|---------------|
| **编排者** | 路由工作，跟踪状态，做出优先级决策 | 高推理能力模型（处理判断） |
| **构建者** | 生产工件 —— 代码、文档、配置 | 机械性工作可以使用成本效益高的模型 |
| **审查者** | 验证质量，对缺陷进行回退 | 高推理能力模型（捕获构建者遗漏的问题） |
| **运维 (Ops)** | Cron 任务、站立会议、健康检查、调度 | 最便宜但可靠的模型 |

→ *定义新团队或添加代理时，阅读 [references/team-setup.md](references/team-setup.md)。*

### 任务状态

每个任务都会经历定义好的生命周期：

```
收件箱 → 已分配 → 进行中 → 审查 → 完成 | 失败
```

**规则：**
- 编排者拥有状态转换权 —— 不要依赖代理自己更新状态
- 每次转换都要有评论（谁、什么、为什么）
- 失败是一个有效的终态 —— 捕获原因并继续

→ *设计任务流或调试卡住的任务时，阅读 [references/task-lifecycle.md](references/task-lifecycle.md)。*

### 移交

当工作在代理之间传递时，移交消息包括：

1. **做了什么** —— 变更/输出的摘要
2. **工件在哪里** —— 精确的文件路径
3. **如何验证** —— 测试命令或验收标准
4. **已知问题** —— 任何未完成或有风险的部分
5. **下一步** —— 接收代理的明确下一步行动

糟糕的移交：*"完成了，检查文件。"*
良好的移交：*"在 `/shared/artifacts/auth/` 构建了认证模块。运行 `npm test auth` 进行验证。已知问题：速率限制尚未实现。下一步：审查者检查错误处理边缘情况。"*

### 审查

跨角色审查防止质量退化：

- **构建者审查规格** —— "这可行吗？缺少什么？"
- **审查者检查构建** —— "这符合规格吗？边缘情况呢？"
- **编排者审查优先级** —— "这是当前正确的工作吗？"

跳过审查步骤，质量会在 3-5 个任务内下降。每次都是如此。

→ *设置代理通信通道时，阅读 [references/communication.md](references/communication.md)。*
→ *实现经过验证的多步骤工作流时，阅读 [references/patterns.md](references/patterns.md)。*

## 常见陷阱

### 派生时没有明确的工件输出路径
代理做出了很好的工作，但你找不到它。总是在派生提示中指定精确的输出路径。使用具有可预测结构的共享工件目录。

### 没有审查步骤 = 质量退化
"这是个小的变更，跳过审查。" 这样做三次，你就会有复合错误。每个工件至少要有一双没有生产它的眼睛来审查。

### 代理不评论任务进度
沉默的代理会造成协调盲点。要求在以下时候评论：开始、阻塞、移交、完成。如果代理沉默，假设它卡住了。

### 分配前不验证代理能力
将基于浏览器的测试分配给没有浏览器访问权限的代理。将图像工作分配给仅文本模型。路由前检查能力。

### 编排者做执行工作
编排者路由和跟踪 —— 它不构建。当你开始"快速做这一件事"的那一刻，你就失去了对团队其他部分的监督。

## 何时不要使用此技能

- **单代理设置** —— 只需遵循标准的 AGENTS.md 约定。团队编排增加了单代理不需要的开销。
- **一次性任务委托** —— 直接使用 `sessions_spawn`。此技能适用于具有多次移交的持续工作流。
- **简单问题路由** —— 如果你只是将问题转发给专家，那是一条消息，而不是工作流。

此技能适用于 **持续团队工作流** —— 代理在多个任务中相互依赖输出的重复协作模式。
